 28: /**
 29:  * A <tt>HashedCredentialMatcher</tt> provides support for hashing of supplied <tt>AuthenticationToken</tt> credentials
 30:  * before being compared to those in the <tt>AuthenticationInfo</tt> from the data store.
 31:  *
 32:  * <p>Credential hashing is one of the most common security techniques when safeguarding a user's private credentials
 33:  * (passwords, keys, etc).  Most developers never want to store their users' credentials in plain form, viewable by
 34:  * anyone, so they often hash the users' credentials before they are saved in the data store.</p>
 35:  *
 36:  * <p>This class (and its subclasses) function as follows:</p>
 37:  *
 38:  * <p>It first hashes the <tt>AuthenticationToken</tt> credentials supplied by the user during their login.  It then
 39:  * compares this hashed value directly with the <tt>AuthenticationInfo</tt> credentials stored in the system.  The stored account
 40:  * credentials are expected to already be in hashed form.  If these two values are equal, the submitted credentials
 41:  * match.</p>
 42:  *
 43:  * <h3>Salting and Multiple Hash Iterations</h3>
 44:  *
 45:  * <p>Because simple hashing is sometimes not good enough for many applications, this class also supports 'salting'
 46:  * and multiple hash iterations.  Please read this excellent
 47:  * <a href="http://www.owasp.org/index.php/Hashing_Java" _target="blank">Hashing Java article</a> to learn about
 48:  * salting and multiple iterations and why you might want to use them. (Note of sections 5
 49:  * &quot;Why add salt?&quot; and 6 "Hardening against the attacker's attack").
 50:  *
 51:  * <p>We should also note here that all of JSecurity's Hash implementations (for example,
 52:  * {@link org.jsecurity.crypto.hash.Md5Hash Md5Hash}, {@link org.jsecurity.crypto.hash.Sha1Hash Sha1Hash}, etc)
 53:  * support salting and multiple hash iterations via overloaded constructors.</p>
 54:  *
 55:  * <h4>Salting</h4>
 56:  *
 57:  * <p>Salting of the authentication token's credentials hash is disabled by default, but you may enable it by setting
 58:  * {@link #setHashSalted hashSalted} to
 59:  * <tt>true</tt>.  If you do enable it, the value used to salt the hash will be
 60:  * obtained from {@link #getSalt(AuthenticationToken) getSalt(authenticationToken)}.
 61:  *
 62:  * <p>The default <tt>getSalt</tt> implementation merely returns
 63:  * <code>token.getPrincipal()</code>, effectively using the user's identity as the salt, a most common
 64:  * technique.  If you wish to provide the authentication token's salt another way, you may override this
 65:  * <tt>getSalt</tt> method.
 66:  *
 67:  * <h4>Multiple Hash Iterations</h4>
 68:  *
 69:  * <p>If you hash your users' credentials multiple times before persisting to the data store, you will also need to
 70:  * set this class's {@link #setHashIterations(int) hashIterations} property.</p>
 71:  *
 72:  * <p><b>Note:</b> <a href="http://en.wikipedia.org/wiki/MD5">MD5</a> and
 73:  * <a href="http://en.wikipedia.org/wiki/SHA_hash_functions">SHA-1</a> algorithms are now known to be vulnerable to
 74:  * compromise and/or collisions (read the linked pages for more).  While most applications are ok with either of these
 75:  * two, if your application mandates high security, use the SHA-256 (or higher) hashing algorithms and their
 76:  * supporting <code>CredentialsMatcher</code> implementations.</p>
 77:  *
 78:  * @author Les Hazlewood
 79:  * @see org.jsecurity.crypto.hash.Md5Hash
 80:  * @see org.jsecurity.crypto.hash.Sha1Hash
 81:  * @see org.jsecurity.crypto.hash.Sha256Hash
 82:  * @since 0.9
 83:  */

226:      * Base64 decode the byte array</li>
