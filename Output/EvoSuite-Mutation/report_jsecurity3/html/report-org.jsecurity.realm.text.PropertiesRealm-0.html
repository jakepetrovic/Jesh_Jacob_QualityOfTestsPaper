 38: /**
 39:  * A subclass of <tt>SimpleAccountRealm</tt> that defers all logic to the parent class, but just enables
 40:  * {@link java.util.Properties Properties} based configuration in addition to the parent class's String configuration.
 41:  *
 42:  * <p>This class allows processing of a single .properties file for user, role, and
 43:  * permission configuration.
 44:  *
 45:  * <p>For convenience, if the {@link #setResourcePath resourcePath} attribute is not set, this class defaults to lookup
 46:  * the properties file definition from <tt>classpath:jsecurity-users.properties</tt> (root of the classpath).
 47:  * This allows you to use this implementation by simply defining this file at the classpath root, instantiating this
 48:  * class, and then calling {@link #init init()}.
 49:  *
 50:  * <p>Or, you may of course specify any other file path using the <tt>url:</tt>, <tt>file:</tt>, or <tt>classpath:</tt>
 51:  * prefixes.</p>
 52:  *
 53:  * <p>If none of these are specified, and the jsecurity-users.properties is not included at the root of the classpath,
 54:  * a default failsafe configuration will be used.  This is not recommended as it only contains a few simple users and
 55:  * roles which are probably of little value to production applications.</p>
 56:  *
 57:  * <p>The Properties format understood by this implementation must be written as follows:
 58:  *
 59:  * <p>Each line's key/value pair represents either a user-to-role(s) mapping <em>or</em> a role-to-permission(s)
 60:  * mapping.
 61:  *
 62:  * <p>The user-to-role(s) lines have this format:</p>
 63:  *
 64:  * <p><code><b>user.</b><em>username</em> = <em>password</em>,role1,role2,...</code></p>
 65:  *
 66:  * <p>Note that each key is prefixed with the token <tt><b>user.</b></tt>  Each value must adhere to the
 67:  * the {@link #setUserDefinitions(String) setUserDefinitions(String)} JavaDoc.</p>
 68:  *
 69:  * <p>The role-to-permission(s) lines have this format:</p>
 70:  *
 71:  * <p><code><b>role.</b><em>rolename</em> = <em>permissionDefinition1</em>, <em>permissionDefinition2</em>, ...</code></p>
 72:  *
 73:  * <p>where each key is prefixed with the token <tt><b>role.</b></tt> and the value adheres to the format specified in
 74:  * the {@link #setRoleDefinitions(String) setRoleDefinitions(String)} JavaDoc.
 75:  *
 76:  * <p>Here is an example of a very simple properties definition that conforms to the above format rules and corresponding
 77:  * method JavaDocs:
 78:  *
 79:  * <code>user.root = <em>rootPassword</em>,administrator<br/>
 80:  * user.jsmith = <em>jsmithPassword</em>,manager,engineer,employee<br/>
 81:  * user.abrown = <em>abrownPassword</em>,qa,employee<br/>
 82:  * user.djones = <em>djonesPassword</em>,qa,contractor<br/>
 83:  * <br/>
 84:  * role.administrator = org.jsecurity.authz.support.AllPermission<br/>
 85:  * role.manager = com.domain.UserPermission,*,read,write;com.domain.FilePermission,/usr/local/emailManagers.sh,execute<br/>
 86:  * role.engineer = com.domain.FilePermission,/usr/local/tomcat/bin/startup.sh,read,execute<br/>
 87:  * role.employee = com.domain.IntranetPermission,useWiki<br/>
 88:  * role.qa = com.domain.QAServerPermission,*,view,start,shutdown,restart;com.domain.ProductionServerPermission,*,view<br/>
 89:  * role.contractor = com.domain.IntranetPermission,useTimesheet</code>
 90:  *
 91:  * @author Les Hazlewood
 92:  * @author Jeremy Haile
 93:  * @since 0.2
 94:  */
