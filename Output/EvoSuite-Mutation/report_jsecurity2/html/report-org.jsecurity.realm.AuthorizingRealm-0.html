 35: /**
 36:  * An <tt>AuthorizingRealm</tt> extends the <tt>AuthenticatingRealm</tt>'s capabilities by adding Authorization
 37:  * (access control) support.
 38:  *
 39:  * <p>This implementation will perform all role and permission checks automatically (and subclasses do not have to
 40:  * write this logic) as long as the
 41:  * {@link #getAuthorizationInfo(org.jsecurity.subject.PrincipalCollection)} method returns an
 42:  * {@link AuthorizationInfo}.  Please see that method's JavaDoc for an in-depth explanation.
 43:  *
 44:  * <p>If you find that you do not want to utilize the {@link AuthorizationInfo AuthorizationInfo} construct,
 45:  * you are of course free to subclass the {@link AuthenticatingRealm AuthenticatingRealm} directly instead and
 46:  * implement the remaining Realm interface methods directly.  You might do this if you want have better control
 47:  * over how the Role and Permission checks occur for your specific data source.  However, using AuthorizationInfo
 48:  * (and its default implementation {@link SimpleAuthorizationInfo SimpleAuthorizationInfo}) is sufficient in the large
 49:  * majority of Realm cases.
 50:  *
 51:  * @author Les Hazlewood
 52:  * @author Jeremy Haile
 53:  * @see SimpleAuthorizationInfo
 54:  * @since 0.2
